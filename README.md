# p0-w3-d1-t9 · Проверка данных регистрации

> Тип: синтез тем дня (побольше) · Репозиторий: один отдельный

> Подключить файл, отсмотреть в браузере.

## Задача

Бэкенд получил данные из формы регистрации и, прежде чем что-то с ними делать, должен их
проверить. Полей пять, у каждого свои правила «годное / не годное». Циклов и функций ещё не
проходили, поэтому данные задаёшь переменными в начале файла, а каждое поле проверяешь по
очереди — это и есть реальная логика валидации, просто записанная в лоб.

```
данные из формы (заданы переменными сверху файла)
  email, имя, возраст, согласие, телефон
          │  проверяем по правилам
          ▼
  по каждому полю: ок / в чём проблема
  итог: можно регистрировать или нет
```

## Вводные

В начале `validate.js` задать переменными пять полей (значения подбери сама, в т.ч. заведомо
плохие, чтобы проверить ветки):

- `email` — строка;
- `name` — строка;
- `age` — должно быть числом;
- `agreed` — согласие с условиями (булево);
- `promo` — промокод, может отсутствовать (`null`/`undefined`/пустая строка).

## Правила проверки

- `email`: не пустой и содержит `@` (достаточно проверки на наличие символа — без регулярок,
  их ещё не было);
- `name`: не пустая строка и не только пробелы;
- `age`: действительно число (не строка `'25'`), и в разумном диапазоне (например 14–120);
- `agreed`: строго `true` (тут важно строгое сравнение, а не truthy — объясни почему);
- `promo`: если не задан — подставить значение по умолчанию `'без промокода'` через `??`
  (подумай, почему именно `??`, а не `||`);
- итог: вывести по каждому полю результат («ок» либо в чём проблема) и общий вердикт —
  можно регистрировать или нет (можно, только если все обязательные поля прошли).

## Критерии приёмки

* По каждому из пяти полей выводится понятный результат проверки и есть общий вердикт.
* `age` проверяется именно на тип «число» (`'25'` не проходит), `agreed` — строгим сравнением
  с `true`; ты можешь объяснить, почему здесь нельзя положиться на truthy.
* Отсутствующий `promo` подставляется через `??`; объяснено, почему не `||`.
* На заведомо плохих данных (пустой email, возраст строкой, нет согласия) вердикт — отказ, и
  видно, какое поле его завалило. Код не падает.
